## 💾 显存需求计算

### 方案1: Qwen3.5-35B-A3B (MoE模型)

**数据来源**: [Qwen3.5-35B-A3B Specifications](https://apxml.com/models/qwen35-35b-a3b)

```
总参数: 35B
激活参数: 3B (每个token)
MoE架构: 256个专家，每次激活8个专家+1个共享专家

显存计算:
1. 模型权重 (FP16):
   - 35B × 2 bytes = 70GB

2. 模型权重 (FP8量化):
   - 35B × 1 byte = 35GB ✅

3. KV Cache (批处理):
   - 每个请求: ~2GB (8K context)
   - 4卡并行: 可支持 (96GB - 35GB) / 2GB = 30个并发请求 ✅

4. 激活值:
   - 每次推理: ~4GB
   - 4卡分摊: 1GB/卡

总显存需求 (4卡Tensor Parallel):
- 模型: 35GB / 4 = 8.75GB/卡
- KV Cache: 15GB/卡 (30并发)
- 激活值: 1GB/卡
- 总计: ~25GB/卡 ⚠️ (接近24GB上限)

优化方案:
- 使用FP8量化: 8.75GB → 4.4GB/卡
- 减少并发: 30 → 20个请求
- 优化后: 4.4 + 10 + 1 = 15.4GB/卡 ✅
```

**结论**: 4卡4090可以运行Qwen3.5-35B-A3B (FP8)，支持20-25个并发请求。

### 方案2: FLUX.2 dev (图像生成)

**数据来源**: [FLUX.2 dev RTX 4090 Performance](https://droid4x.com/best-gpu-for-stable-diffusion-sdxl-and-flux/)

```
模型大小: ~12GB (FP16)
单张图片生成:
- 分辨率: 1024x1024
- 步数: 20步
- 时间: 2-3秒 (RTX 4090)
- 显存占用: ~12GB

批处理 (Batch Size):
- Batch=1: 12GB显存，3秒/张
- Batch=2: 18GB显存，5秒/2张 (2.5秒/张)
- Batch=4: 无法运行 (超过24GB)

4卡并行:
- 4个独立任务同时生成
- 每卡Batch=1
- 总吞吐: 4张/3秒 = 80张/分钟 = 4800张/小时

加速方案 (Turbo LoRA):
- 步数: 20 → 8步
- 时间: 3秒 → 1秒/张
- 显存: 12GB (不变)
- 总吞吐: 4张/秒 = 240张/分钟 = 14,400张/小时 ✅
```

**结论**: 4卡4090 + Turbo LoRA可以达到14,400张/小时，远超需求。

## ⚠️ 潜在瓶颈分析

### 1. 显存瓶颈 (已解决)

```
问题: Qwen3.5-35B-A3B FP16需要70GB
解决: FP8量化 → 35GB，4卡分摊 → 8.75GB/卡 ✅

问题: 并发请求占用KV Cache
解决: 限制并发20-25个，预留15GB/卡 ✅
```

### 2. 带宽瓶颈 (需注意)

```
PCIe 4.0 x16带宽: 32 GB/s
4卡Tensor Parallel通信:
- 每次推理: ~2GB数据传输
- 200 tokens/s: 2GB × 200 / 500 = 0.8 GB/s
- 占用率: 0.8 / 32 = 2.5% ✅

建议: 使用NVLink (如果主板支持)
- 带宽: 600 GB/s
- 但RTX 4090不支持NVLink
- PCIe 4.0足够
```

### 3. 存储瓶颈 (需优化)

```
MinIO写入需求:
- 8000套 × 8张 × 500KB = 32GB/天
- 写入速度: 32GB / 86400秒 = 0.37 MB/s ✅

NVMe SSD性能:
- 顺序写入: 3000 MB/s
- 随机写入: 500 MB/s
- 完全满足需求 ✅

建议: RAID 0 (2 × 2TB) 提升性能
```

### 4. 网络瓶颈 (可选优化)

```
当前需求:
- 内部通信: GPU间、服务间
- 外部API: 1688、平台API
- 带宽需求: < 100 Mbps

1GbE网卡: 完全满足 ✅
10GbE网卡: 未来扩展 (可选)
```
