> AI Native 跨境卖家操作系统 + ERP Lite 内核
>
> 版本: v1.0

> 适用对象: 创始人、产品、业务、投资人

---

## 📌 一句话定位

**Deyes 是一个 AI Native 的跨境卖家操作系统，让卖家以 SKU 为核心，在一个系统里完成：选品、找供应商、算利润、控风险、生成素材、多平台上架、跟踪经营结果，并让系统自动把结果反哺下一轮决策。**

---

## 🎯 北极星指标

### 核心价值主张

传统跨境卖家的痛点：
- 选品靠经验，成功率低
- 测款周期长，成本高
- 素材制作慢，质量不稳定
- 多平台管理混乱
- 经营数据分散，无法闭环优化

Deyes 的解决方案：
- **AI 选品**：从历史数据和市场趋势中发现高潜力商品
- **AI 内容**：自动生成多风格主图和详情页，支持 A/B 测试
- **多平台统一**：一次上架，多平台同步，统一管理
- **闭环优化**：经营结果自动反哺选品和定价决策
- **ERP Lite 内核**：商品、库存、采购、订单、利润统一管理

### 目标用户

**主要用户**：
- 跨境电商卖家（Temu / Amazon / Ozon / AliExpress）
- 日测款需求 50-200 SKU
- 追求快速验证和数据驱动决策
- 希望降低人力成本和提升运营效率

**次要用户**：
- 供应链服务商
- 跨境电商代运营公司
- 品牌出海团队

### 成功指标

**Stage 1-2 (可经营闭环)**：
- 单个 SKU 从发现到上架 < 2 小时
- 支持 3-5 个主图风格变体
- 真实平台表现数据回流系统
- 用户能看到 CTR / CVR / 销售额 / 退款率

**Stage 3 (经营反馈引擎)**：
- 历史高表现 seed 优先进入 recall
- 系统能解释为什么推荐某个商品
- 低转化风格自动降权
- 高退款供应商自动标记

**Stage 4 (ERP Lite 核心)**：
- 单个 SKU 可追溯：候选 → 供应商 → 采购 → 库存 → 订单 → 利润
- 库存自动同步到平台
- 采购单自动生成
- 实际净利可计算

**Stage 5 (多平台经营中枢)**：
- 支持 3+ 平台统一管理
- 多币种自动转换
- 多语言内容自动生成
- 平台策略差异化配置

**Stage 6 (自动化经营)**：
- 80% 日常动作自动执行（调价、补货、下架、换素材）
- 运营者只处理例外和审批
- 商品生命周期自动管理

## 🧭 战略定位

### 我们不是什么

❌ **不是通用 ERP**
我们不做总账、财务报表、人事、CRM、传统进销存大杂烩。

❌ **不是单点工具**
我们不只是"图片生成工具"或"选品工具"，而是完整的卖家操作系统。

❌ **不是纯 SaaS**
我们优先服务自营或深度合作卖家，而不是追求海量小 B 用户。

### 我们是什么

✅ **AI Native 跨境卖家操作系统**
决策层（AI Agent）+ 执行层（自动化）+ 事实层（ERP Lite）+ 反馈层（闭环优化）

✅ **数据驱动的测款引擎**
快速测试 → 数据验证 → 爆款放大 → 滞销清退

✅ **多平台经营中枢**
统一商品、库存、价格、素材、表现，跨平台协同

## 🗺️ 分阶段路线图

### Stage 0：固化基线，区分核心与集成

**目标**：把当前已跑通的能力固化成可持续开发基线

**关键交付**：
- 测试分层（unit / regression / integration）
- 环境分层（开发 / 测试 / 联调 / 生产）
- 可观测性（全链路追踪）

**业务价值**：
- 稳定的开发基础
- 快速定位问题
- 支撑后续迭代

**退出标准**：
- 核心回归测试稳定通过
- 外部集成测试单独运行
- 一次任务可完整追踪

---

### Stage 1：可经营闭环

**目标**：从"能选品、能上架"升级为"能看结果、能复盘、能持续优化"

**关键交付**：
1. **平台表现数据回流**
   - 曝光 / 点击 / CTR / 订单 / 销售额 / 退款 / 退货

2. **Listing 表现中心**
   - 每日 metrics / 累计 metrics
   - 按 listing / 素材 / 平台维度聚合

3. **内容 A/B 测试基础**
   - 多风格变体生成
   - 表现归因
   - 赢家识别

4. **真实平台同步**
   - Temu / Amazon API 集成
   - 库存 / 价格实时同步

**业务价值**：
- 从"盲测"到"数据驱动"
- 快速识别爆款
- 及时止损滞销品

**退出标准**：
- 一个商品从发现到上架后，能拿到真实经营结果
- 能按 listing / 素材 / 平台看表现
- 结果能回流系统

---

### Stage 2：经营反馈引擎

**目标**：把 Phase 6 从"先验加分"升级为真正的经营闭环

**关键交付**：
1. **Feedback Aggregator 升级**
   - 不止影响 seed / shop / supplier
   - 还影响 style / asset pattern / price band / region / platform

2. **历史反馈进入 recall + ranking**
   - 高表现 seed 优先 recall
   - 历史先验影响 business score

3. **解释性输出**
   - 系统能解释为什么推荐某个商品
   - 可视化历史表现

4. **淘汰机制**
   - 高退款 seed 降权
   - 高侵权 risk 组合标记
   - 低转化风格降权
   - 高售后 supplier 标记

**业务价值**：
- 系统越用越聪明
- 减少重复踩坑
- 提升整体成功率

**退出标准**：
- 新一轮候选排序明显受真实经营结果影响
- 人能解释为什么系统偏爱某类 seed / supplier / style
- 不再是纯静态启发式

---

### Stage 3：ERP Lite 核心

**目标**：建立最小但够用的 ERP Lite 事实层

**关键交付**：
1. **商品主数据中心**
   - ProductMaster / SPU / SKU
   - 规格属性
   - Candidate → ProductMaster 转化

2. **Listing 中心**
   - 每个平台 listing 的状态中心
   - Listing ↔ SKU 映射
   - 资产组合 / 价格策略

3. **库存中心**
   - 可用 / 锁定 / 在途 / 安全库存
   - 平台同步库存

4. **采购中心**
   - 供应商实体
   - 采购单 / 到货单
   - 交期 / MOQ / 采购价历史

5. **订单与售后中心**
   - 平台订单 / 订单项
   - 发货状态 / 退款 / 退货
   - 售后原因

6. **利润台账**
   - 销售收入 / 平台佣金 / 支付费
   - 物流成本 / 广告成本 / 退款损失
   - 实际净利

**业务价值**：
- 统一事实层
- 可追溯可审计
- 支撑自动化决策

**退出标准**：
对一个 SKU，系统能回答：
- 它从哪个候选商品演化而来
- 绑定了哪个供应商
- 采购价多少 / 库存多少
- 上了哪些平台 / 卖了多少
- 退款多少 / 净利多少

---

### Stage 4：多平台经营中枢

**目标**：从单平台闭环走向多平台统一经营

**关键交付**：
1. **平台适配扩展**
   - Amazon / Ozon / AliExpress / Rakuten / Mercado Libre

2. **统一 listing policy**
   - 平台类目映射
   - 平台禁售规则
   - 平台价格策略
   - 平台素材规范

3. **多币种与地区化**
   - 汇率自动转换
   - 税费计算
   - 地区利润线
   - 地区退货风险

4. **多语言与本地化内容**
   - 标题 / 卖点 / 详情页
   - 风险词与敏感词控制

**业务价值**：
- 多平台统一管理
- 降低运营复杂度
- 扩大市场覆盖

**退出标准**：
同一 SKU 能在多个平台统一管理：
- 上架状态 / 库存 / 价格
- 表现 / 利润

---

### Stage 5：自动化经营与策略控制台

**目标**：让系统不止看数据，而是真正开始自动经营

**关键交付**：
1. **商品生命周期控制**
   - 测试期 / 放量期 / 稳定期 / 衰退期 / 清退期

2. **自动动作引擎**
   - 调价 / 补货 / 下架 / 换素材
   - 重新选 supplier / 扩平台 / 降权 seed

3. **经营控制台**
   - 哪些 SKU 该加码
   - 哪些 SKU 该清退
   - 哪些异常需要人工处理

4. **异常检测**
   - CTR 异常下降 / 退款异常上升
   - 库存断货风险 / 供应商履约风险
   - 平台风控风险

**业务价值**：
- 运营者从"手工做动作"变成"审批例外、修正规则、看经营结果"
- 大幅降低人力成本
- 提升响应速度

**退出标准**：
- 80% 日常动作自动执行
- 运营者只处理例外
- 商品生命周期自动管理

## ⚠️ 关键风险与应对

### 风险 1：平台 API 限流

**影响**：无法大规模发布和同步

**应对**：
- 实现指数退避和请求队列
- 多账号轮换
- 混合策略（80% API + 20% RPA）

---

### 风险 2：图片质量不稳定

**影响**：低 CTR，品牌形象受损

**应对**：
- 实现质量评分
- 人工审核关键素材
- A/B 测试验证

---

### 风险 3：库存断货

**影响**：Listing 缺货，流失销售

**应对**：
- 安全库存缓冲
- 自动补货提醒
- 供应商交期追踪

---

### 风险 4：汇率波动

**影响**：利润侵蚀

**应对**：
- 每日汇率更新
- 动态调价
- 利润预警

---

### 风险 5：GPU 集群瓶颈

**影响**：无法扩容

**应对**：
- 横向扩展（增加 GPU 节点）
- 请求队列
- 混合云（80% 本地 + 20% API）

## 🎯 为什么这个顺序最合理

### 1. 先有真实结果，反馈才有意义
如果没有 listing 表现、订单、退款、真实利润，Phase 6 只能是"伪闭环"。

### 2. 先做 ERP Lite，而不是大 ERP
因为核心竞争力不是传统管理软件，而是：
- 更快找到能卖的货
- 更快生成能卖的素材
- 更快多平台验证
- 更快把经营结果反哺下一轮

### 3. 先单闭环跑顺，再多平台扩张
如果 Temu / 1688 这一条链路还没彻底跑顺，多平台扩张只会把复杂度成倍放大。

### 4. 先事实层，再全自动
没有统一事实层，自动化只会把错误放大。

## 📝 附录：术语表
